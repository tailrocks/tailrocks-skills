# Handoff and Index

How the `tailrocks-blueprint` skill wires the blueprint together so any fresh
executor — a new session of the same agent, a cheaper model, or a different
agent product entirely — can pick up with zero context.

## Plans manifest and index: `blueprints/<slug>/plans/README.md`

Written **before** the plan-writer subagents run (it is their input), finalized
after review, updated by executors as they work:

```markdown
# Implementation Plans — <slug>

Generated by the tailrocks-blueprint skill on <date> at commit `<short SHA>`,
from concept `<path>`. Execute in the order below unless dependencies say
otherwise. Each executor: read the plan fully before starting, run its
preconditions first, honor its STOP conditions, and update your row when done.

## Execution order & status

| Plan | Title | Covers | Priority | Effort | Depends on | Status |
|------|-------|--------|----------|--------|------------|--------|
| 001  | ...   | F1, S1 | P1       | M      | —          | TODO   |
| 002  | ...   | F3     | P1       | M      | 001        | TODO   |

Status values: TODO | IN PROGRESS | DONE | BLOCKED (one-line reason) |
REJECTED (one-line rationale) | STALE (re-run pending; see spec change log)

## Item briefs

### 001 — <title>
- **Goal**: <one sentence>
- **Covers**: <requirement headings + ledger IDs>
- **Scope**: in — <areas>; out — <areas, incl. plans that own them>
- **Guardrails**: <N# IDs>
- **Research**: <chapters>

## Dependency notes

- 002 requires 001 because <reason>.

## Deferrals carried from the spec

- <ID>: <why it is safe to defer, and what triggers revisiting>.
```

The item briefs are the contract handed to each plan-writer subagent; keeping
them in the manifest means the decomposition survives any context reset and the
review gate can check plans against exactly what was commissioned.

## Blueprint hub: `blueprints/<slug>/README.md`

The front door. Contents, in order:

- **Status**: DRAFTING | RESEARCHED | SPECIFIED | PLANNED | IN EXECUTION —
  with date and commit.
- **Concept**: path to the concept file (the human's document — never edited).
- **Artifacts**: links to `coverage.md`, `research/00-summary.md`, `spec/README.md`,
  `plans/README.md`.
- **Executor protocol** (verbatim, so no executor has to be told):

```markdown
## Executor protocol

One plan per fresh session. For each plan, in order from `plans/README.md`:

1. Read the plan file fully. Run its preconditions; a failure is a STOP.
2. Follow the steps; run every verification; honor every STOP condition and
   Must NOT.
3. On completion, check every done criterion, then update the plan's status
   row in `plans/README.md`.
4. On a STOP: set the row to BLOCKED with a one-line reason and report back.
   Do not start the next plan on top of a BLOCKED dependency.

Plans are self-contained — do not read the concept, spec, or research to fill
gaps; a gap is a plan defect to report, not to improvise around.
```

- **Next Agent Prompt** (always last, always current):

```markdown
## Next Agent Prompt

> **Status**: PLANNED — <N> plans ready under `plans/`, none started.
> **Pick up at**: `plans/001-<slug>.md`. Read it fully, run its preconditions
> first, honor its STOP conditions.
> **Then**: proceed in the order in `plans/README.md`; update each status row.
> **Blockers**: <none | the input X from plan NNN's "Inputs to provide">.
> **Global checklist**:
> - [ ] 001-<slug> — <one line>
> - [ ] 002-<slug> — <one line>
```

The protocol and prompt are written agent-neutrally — no product-specific
commands or namespaces — so the same blueprint drives whichever executor the
human points at it.

## Reconcile on re-entry

When the skill is re-invoked on an existing blueprint, before any new work:

- **DONE** rows: spot-check the cheap done criteria still hold; note verified.
- **BLOCKED** rows: read the reason, investigate, then refresh the plan
  (in-place for detail fixes, new number when the approach changed) or mark
  REJECTED with one line.
- **STALE** rows (from a spec change log entry): refresh the plan against the
  updated requirement text and clear the flag.
- **TODO** rows: run the plan's preconditions and drift check; refresh
  "Starting state" excerpts and the planned-at SHA if the world moved.

Then update the hub status and rewrite the Next Agent Prompt. Plan files are
never deleted — they are the record.

## Why this shape

- The **manifest-first** order means plan writing is commissioned work against
  a reviewed decomposition, not improvisation by parallel subagents.
- The **hub** removes the "where do I start?" gap that costs a fresh agent its
  first several tool calls.
- Everything — concept pointer, ledger, evidence, spec, plans, status — travels
  in one folder and survives any context reset.
